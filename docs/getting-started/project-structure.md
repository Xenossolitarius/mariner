# Project Structure

This page explains the key files and directories in a Mariner project and how they relate to each other.

## Key Files

### `mariner.config.ts`

The configuration file for a microfrontend app. Every app must have one. Mariner scans your project for these files to discover apps automatically.

```ts
import { defineMarinerConfig } from 'mariner-fe'

export default defineMarinerConfig({
  mariner: 'my-app',
  plugins: [],
})
```

The `mariner` field is the app's unique identifier. It determines the URL path for the navigator module and is used as the key when other apps import from this one.

### `navigator.ts`

The public entry point of a microfrontend. This file exports the `navigator` object, which is what other apps and host pages consume.

```ts
import { createVueNavigator } from 'mariner-fe/vue'
import { createApp } from 'vue'
import App from './App.vue'

const app = createApp(App)
export const navigator = createVueNavigator(app)
```

::: tip
The navigator file must be a sibling of `mariner.config.ts` in the same directory. Mariner uses the config file location to find the navigator.
:::

### `cargo.ts` (Optional)

A cargo file allows an app to declare shared data or dependencies that it provides to other apps. Think of it as a secondary export surface, separate from the navigator.

```ts
// cargo.ts
export const sharedState = reactive({ count: 0 })
```

Other apps can import cargo using the `cargo:` virtual module syntax, similar to how `navigator:` imports work.

### `fleet.config.json` (Optional)

A fleet configuration groups multiple apps together for coordinated development and builds. This file is placed at the project root.

```json
{
  "apps": ["app1", "app2", "shared"]
}
```

When you run `mariner dev` or `mariner build` in fleet mode, all listed apps are started or built together.

## Generated Directories

### `.mariner/`

Each app gets a `.mariner/` directory containing auto-generated TypeScript type definitions. These provide type safety for virtual module imports like `navigator:shared`.

::: warning
Do not edit files inside `.mariner/` manually. They are regenerated by the framework and your changes will be overwritten.
:::

## Monorepo Scanning

In a monorepo setup, Mariner automatically discovers all apps by scanning for `mariner.config.ts` (or `.js`) files across the workspace. You do not need to register apps manually -- placing a config file in a directory is enough for Mariner to recognize it as an app.

A typical monorepo layout looks like this:

```
project-root/
  fleet.config.json
  pnpm-workspace.yaml
  packages/
    shared/
      mariner.config.ts
      navigator.ts
      .mariner/           <-- generated types
    app1/
      mariner.config.ts
      navigator.ts
      cargo.ts            <-- optional
      .mariner/           <-- generated types
    app2/
      mariner.config.ts
      navigator.ts
      .mariner/           <-- generated types
```

## How Files Relate

The relationship between these files follows a clear pattern:

1. **`mariner.config.ts`** declares the app and its build configuration (name, Vite plugins, options).
2. **`navigator.ts`** provides the runtime entry point that other apps and host pages consume.
3. **`cargo.ts`** optionally exposes shared data or utilities alongside the navigator.
4. **`fleet.config.json`** ties multiple apps together so they can be developed and built as a group.
5. **`.mariner/`** is generated from the fleet configuration, providing TypeScript definitions for all cross-app imports.

When Mariner starts, it scans for config files, resolves the fleet, generates types into each app's `.mariner/` directory, and spins up isolated Vite dev servers for each app.
